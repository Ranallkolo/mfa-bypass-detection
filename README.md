# MFA Bypass Detection Framework

An AI-driven framework for detecting and responding to Multi-Factor Authentication (MFA) bypass attack patterns — a proof-of-concept built for an undergraduate thesis at the Air Force Institute of Technology (AFIT), Kaduna.

The system fuses a Random Forest and an LSTM model with contextual reputation signals and behavioural heuristics into a single risk score, then routes each login event to **Allow**, **Step-Up MFA**, or **Block** based on that score. It targets four MFA bypass patterns: SIM swap, adversary-in-the-middle (AiTM) phishing, session hijacking, and MFA fatigue.

## Results Summary

Evaluated on 40,000 held-out test events after noise-injected retraining:

| Metric | Random Forest | LSTM | Full Framework |
|---|---|---|---|
| Accuracy | 97.14% | 96.45% | 96.80% |
| Recall | 96.54% | 95.86% | 96.53% |
| Precision | 97.72% | 97.01% | 97.04% |
| F1-Score | 97.13% | 96.43% | 96.79% |
| AUC-ROC | 0.9968 | 0.9958 | 0.9934 |
| FPR | 2.25% | 2.96% | 2.94% |

The framework also introduces the **Unnecessary Step-Up Rate (USR)** — a novel usability metric measuring authentication friction imposed on legitimate users. Result: **2.84%**, against an 8% target.

Out-of-distribution evaluation (independently seeded test set) achieved an F1-score of 97.53%, closely matching in-distribution performance.

Mean prediction latency: **42.41 ms** (P99: 47.50 ms), well under the 500 ms target, benchmarked on an Azure `Standard_DS3_v2` VM (2 vCPUs, 7.7 GB RAM).

Full results, ablation study, and feature importance analysis are in the thesis (Chapter 4).

## Architecture

The system is organised into five layers:

1. **Data Collection** — RBA Login Dataset (Wiefling et al., 2020, via Kaggle) plus 100k synthetic attack records covering the four bypass types.
2. **Feature Extraction** — 15 engineered features (temporal, device, network reputation, behavioural), normalised with `StandardScaler`.
3. **Anomaly Detection Engine** — a Random Forest (100 trees, depth 15) and an LSTM (2 layers, 64 units, sequence length 10), each trained independently on 160,000 balanced events.
4. **Decision Engine** — fuses model outputs with contextual and behavioural scores into a single risk score `R`:

   ```
   R = (0.6 × ML + 0.2 × Contextual + 0.2 × Behavioural) × 100
   ML = 0.6 × P_RF + 0.4 × P_LSTM
   ```

   | Tier | Score Range | Action |
   |---|---|---|
   | Allow | R < 34 | Login proceeds, JWT issued |
   | Step-Up MFA | 34 ≤ R < 67 | TOTP challenge required |
   | Block | R ≥ 67 | Access denied, alert raised |

5. **Reporting Dashboard** — a FastAPI application with live audit logging (SQLite), decision distribution and risk-score charts, a manual event simulator, and six preset attack-scenario demos.

## Dataset

This repo does **not** include the underlying dataset due to size:

- `rba-dataset.csv` (~8.5 GB) — the raw RBA Login Dataset (Wiefling et al., 2020). Available from [Kaggle](https://www.kaggle.com/) (search "RBA Dataset" / Wiefling).
- `data/full_dataset.csv`, `data/train_test_split_noisy.npz` — processed/derived files generated by `1_preprocess.py` and `9_retrain_with_noise.py` from the raw dataset above.

To regenerate them locally, download the raw dataset, place it at the repo root as `rba-dataset.csv`, and run the pipeline scripts in numerical order (see **Usage** below).

The trained models (`models/*.pkl`, `models/*.pt`) are included in this repo, so the dashboard and prediction endpoint can be run immediately without regenerating the dataset.

## Usage

**Run the dashboard** (uses the pretrained models already included in the repo):

```bash
uvicorn api.main:app --host 0.0.0.0 --port 8000
```

Then open `http://localhost:8000` (or `http://YOUR_SERVER_IP:8000` if running remotely).

**Test the prediction endpoint directly:**

```bash
curl -X POST http://localhost:8000/predict \
  -H "Content-Type: application/json" \
  -d '{"hour":3,"is_night":1,"is_attack_ip":1,"country_changed":1,"asn_attack_rate":0.9}'
```

**Regenerate results from scratch** (requires the raw dataset — see **Dataset**):

```bash
python 1_preprocess.py
python 2_synthetic.py
python 3_train_rf.py
python 4_train_lstm.py
python 9_retrain_with_noise.py
python 5_evaluate.py
python 6_latency.py
```

## Limitations

- The RBA dataset is primarily IP-reputation-based rather than a confirmed MFA-bypass-specific dataset; the top three features (by importance) are reputation signals, not MFA-specific behavioural signals.
- No live identity provider integration — decisions are demonstrated via the dashboard, not enforced against a real authentication system.
- The dataset is balanced 50/50 between attack and normal events, which does not reflect real-world attack prevalence (typically well under 0.1%); precision would be expected to degrade at production scale.
- Latency was measured on a single, unloaded instance; concurrent-load benchmarking is recommended before any production use.

A full discussion of limitations and future work is in Chapter 5 of the thesis.
